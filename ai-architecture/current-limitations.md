# Current Limitations — Анализ текущей реализации

---

## Сильные стороны

### 1. Чистая MCP-архитектура инструментов
Все инструменты зарегистрированы в одном месте (`McpService.buildToolRegistry()`), имеют JSON Schema, примеры, теги и auth-политики. Это позволяет использовать их одновременно из внутреннего агента и внешних Claude-интеграций без дублирования логики.

### 2. Confirmation flow
Система требует явного подтверждения пользователя для всех write/dangerous операций. Это правильное UX-решение, предотвращающее нежелательные изменения данных.

### 3. Mode-based context
Агент получает специализированный контекст (режим, дату, слот, существующие элементы) от frontend. Это позволяет давать более точные ответы без необходимости AI самостоятельно «выяснять» контекст.

### 4. Валидация через DTO + JSON Schema
Двухуровневая валидация (JSON Schema coercion + class-validator DTO) обеспечивает надёжную защиту domain services от невалидных данных AI.

### 5. Tool policy enforcement
`AgentToolPolicyService` с hardcoded allowlists предотвращает выход AI за пределы разрешённых действий для текущего контекста (например, нельзя удалить рецепт из meal_plan_slot).

### 6. Идемпотентность product creation
Кэш на 30 секунд по сигнатуре продукта предотвращает дублирование при повторных вызовах AI.

## Слабые стороны и технический долг

### 1. Хранение диалогов в памяти процесса

**Проблема**: `AgentConversationService` хранит все диалоги в `Map<string, StoredAgentConversation>` в памяти Node.js процесса.

**Последствия**:
- При рестарте Cloud Run сервиса (деплой, масштабирование, OOM kill) — все диалоги теряются
- Невозможно масштабировать горизонтально (несколько инстанций не видят диалоги друг друга)
- Нет TTL: старые диалоги накапливаются и занимают память бесконечно
- История не персистируется между сессиями пользователя

**Масштаб**: Критично для production при любом горизонтальном масштабировании.

---

### 2. Дублирование агентной логики (backend + frontend)

**Проблема**: Существуют два независимых агентных цикла:
- Backend: `AgentService.runOpenAiLoop()` (primary path)
- Frontend: `openaiAgentApi.runAgentTurn()` (legacy path)

Каждый имеет свои промпты, свою tool execution логику, свои fallback тексты.

**Последствия**:
- Любое изменение поведения нужно делать дважды
- Frontend промпты и backend промпты расходятся со временем
- Quota enforcement (AI usage) может обходиться при прямом OpenAI flow с frontend

**Масштаб**: Средний технический долг. Фактически frontend path используется редко, но остаётся в коде.

---

### 3. Монолитный агент без специализации

**Проблема**: Один `AgentService` обрабатывает все 9 режимов. Логика разветвляется через switch/case в `AgentPromptService` и `applyStructuredResponse()`.

**Где накапливается сложность**:
- `applyStructuredResponse()` — сейчас 80 строк if/else по mode, будет расти
- `AgentToolPolicyService.ALLOWED_TOOLS_BY_MODE` — статичный hardcoded объект
- Промпты для разных режимов живут в одном модуле

**Последствия**:
- Сложно добавить новый режим без затрагивания существующей логики
- Тестировать отдельные сценарии сложно (всё в одном классе)
- Нет возможности независимо оптимизировать поведение для конкретного домена

---

### 4. Короткая история диалога (8 сообщений)

**Проблема**: В `buildInitialInput()` используется `conversation.messages.slice(-8)`. Хранится максимум 12.

**Последствия**:
- Агент теряет контекст разговора при длинных сессиях
- Нельзя ссылаться на решения, принятые ранее в диалоге
- При работе с рецептом из 15 шагов редактирование может "забыть" начальные требования

---

### 5. Ограниченный цикл (max 4 шага)

**Проблема**: `runOpenAiLoop` выполняет максимум 4 итерации.

**Последствия**:
- Сложные многошаговые задачи не выполняются (например: "создай рецепт из найденных продуктов и добавь его в план")
- Агент вынужден либо срезать задачу, либо просить пользователя продолжить вручную

---

### 6. Хрупкий парсинг JSON-ответов AI

**Проблема**: `parseJsonObject()` использует regex и string manipulation для извлечения JSON из текста AI.

```typescript
const fenced = text.match(/```(?:json)?\s*([\s\S]*?)```/i);
const firstBrace = text.indexOf("{");
const lastBrace = text.lastIndexOf("}");
```

**Последствия**:
- Если AI нарушает формат (markdown, несколько JSON-объектов) — данные теряются
- Нет structured outputs от OpenAI → риск hallucinated fields в proposal/draft
- Отсутствует fallback для неполных данных

---

### 7. Модель захардкожена

**Проблема**: `const model = "gpt-5-mini"` — строка захардкожена в `agent.service.ts:44`.

**Последствия**:
- Нельзя переключить модель без деплоя
- Нельзя использовать разные модели для разных режимов (например, сложный recipe draft → gpt-5, простой поиск → gpt-5-mini)
- Эксперименты с моделями затруднены

---

### 8. Нет наблюдаемости (observability)

**Проблема**: Нет трейсинга агентных шагов. Только `console.warn` при ошибках MCP.

**Последствия**:
- Невозможно отлаживать цепочки вызовов (сколько итераций цикла, какие инструменты вызывались, почему агент остановился)
- Нет метрик latency per tool, per mode
- Нельзя строить аналитику по использованию AI-возможностей
- При инцидентах в production — нет данных для постмортема

---

### 9. Confirmation flow — один pending на диалог

**Проблема**: `StoredAgentConversation.pendingConfirmation` хранит только одно ожидающее подтверждение.

**Последствия**:
- Если пользователь задал новый вопрос до подтверждения — pending теряется
- Нельзя запомнить несколько независимых pending actions
- При multi-step планировании ("сначала создай рецепт, потом добавь в план") нельзя держать оба ожидания

---

### 10. Нет семантической памяти о пользователе

**Проблема**: Агент не помнит предпочтения пользователя между сессиями кроме того, что хранится в `UserProfile`.

**Последствия**:
- "Я не ем молочное" нужно повторять в каждом диалоге
- Агент не знает, что пользователь обычно пьёт протеиновый коктейль на завтрак
- История прошлых диалогов не влияет на будущие предложения

---

### 11. Синхронный request/response, нет streaming

**Проблема**: Вся обработка (`POST /v1/agent/chat`) синхронная — frontend ждёт полного ответа.

**Последствия**:
- При multi-step tool use (4 итерации × latency OpenAI + DB) ответ может занимать 10-20 секунд
- Нет UX-обратной связи пока агент "думает"
- Timeout риски при длинных цепочках

---

### 12. Авто-обогащение аргументов — неявная логика

**Проблема**: `enrichToolArguments()` тихо добавляет `date` из контекста в аргументы tool calls.

```typescript
if (selectedDate && args.date == null && ["mealPlan.addEntry", ...].includes(toolName)) {
  return { ...args, date: selectedDate };
}
```

**Последствия**:
- Логика скрыта, не очевидна при отладке
- При добавлении новых инструментов легко забыть добавить их в список

---

### 13. Нет тестов агентной логики

**Проблема**: Unit-тесты (`test/` директория) не покрывают `AgentService`, `AgentPromptService`, `AgentToolPolicyService`.

**Последствия**:
- Изменения в промптах не верифицируются автоматически
- Регрессии в tool policy незаметны до production

---

## Узкие места масштабирования

| Компонент | Узкое место | Решение |
|-----------|-------------|---------|
| Диалоги в памяти | Нет horizontal scaling | Redis / DB persistence |
| Sync request/response | UX latency до ~20с | SSE streaming |
| Единый агент | Сложность растёт линейно | Multi-agent architecture |
| Hardcoded tool policy | Ручное обновление при добавлении фичи | Конфигурационный подход |
| Один pending confirmation | Потеря контекста | Multi-checkpoint state |
