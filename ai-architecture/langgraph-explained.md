# Как работает LangGraph — простым языком

## Что такое LangGraph?

Представь, что раньше агент работал как один большой повар, который делает всё сам: читает заказ, думает, звонит поставщикам, готовит еду, сервирует — и всё это в одном гигантском куске кода.

**LangGraph** — это способ разбить работу агента на шаги (узлы) и соединить их в граф (схему). Как конвейер на заводе: каждая станция делает своё дело, передаёт результат дальше.

---

## Ключевые понятия

### Узел (Node)
Функция, которая что-то делает с данными и возвращает обновлённое состояние.

```
[Роутер] → [Агент для meal plan] → КОНЕЦ
```

### Граф (Graph)
Набор узлов + связи между ними. LangGraph строит граф и запускает узлы в нужном порядке.

### Состояние (State)
Общий объект данных, который передаётся от узла к узлу. Каждый узел может его читать и дополнять.

```
State = {
  messages: [...],          // история диалога
  route: "meal_plan",       // куда роутер решил идти
  pendingConfirmation: ..., // действие ожидает подтверждения
  response: { answer: "..." }
}
```

### Рёбра (Edges)
Связи между узлами. Могут быть:
- **Фиксированные**: A → B всегда
- **Условные**: A → B или C или D — в зависимости от состояния

### Чекпоинтер (Checkpointer)
После каждого узла LangGraph сохраняет состояние в базу данных. Это даёт:
- Персистентную историю диалога (не теряется при рестарте сервера)
- Уникальный `thread_id` = ID разговора с конкретным пользователем

## Наш граф — схема

```
                    ┌─────────────────────────────────┐
                    │           СТАРТ                  │
                    └────────────────┬────────────────┘
                                     ↓
                              ┌─────────────┐
                              │   РОУТЕР    │  ← смотрит на mode и сообщение
                              └──────┬──────┘
                                     │
              ┌──────────────────────┼──────────────────────┐
              │            │         │          │            │
              ↓            ↓         ↓          ↓            ↓
         [recipe]   [meal_plan]  [shopping]  [self_care]  [agent]
              │            │         │          │            │
              └────────────┴─────────┴──────────┴────────────┘
                                     ↓
                                   КОНЕЦ
```

**Каждый нижний узел** — специализированный агент с:
- своим системным промптом
- своим набором разрешённых инструментов
- своей логикой

## Как работает один запрос пользователя — шаг за шагом

### Шаг 1: Получение запроса
Пользователь пишет "добавь яйца на завтрак". Запрос приходит на `POST /v1/agent/chat/stream`.

### Шаг 2: Старт SSE-стрима
Backend открывает SSE-соединение (поток событий) и сразу шлёт `agent_start`. Пользователь видит что агент начал работать.

### Шаг 3: Роутер решает куда идти
```
RouterNode:
  - если mode = "meal_plan_page" → route = "meal_plan"
  - если mode = "global" → спрашивает GPT что имел в виду пользователь
  - если mode = "recipe_create" → route = "recipe"
  - ...
```

### Шаг 4: Специализированный агент обрабатывает запрос
Например, `MealPlanAgentNode`:
1. Формирует системный промпт (с датой, слотами и т.д.)
2. Запускает OpenAI с разрешёнными инструментами
3. OpenAI решает вызвать `mealPlan.addEntry`
4. Backend выполняет инструмент → добавляет яйца в план
5. Результат инструмента отправляется обратно в OpenAI
6. OpenAI формирует финальный ответ пользователю

На каждом шаге 3 и 5 backend шлёт SSE события:
- `tool_call` → фронт показывает "загружаю данные..."
- `tool_result` → инструмент завершён
- `text_delta` → текст ответа приходит кусочками в реальном времени

### Шаг 5: Подтверждение (если нужно)
Если действие опасное (удалить, добавить несколько вещей), агент не выполняет сразу. Он создаёт `PendingConfirmation` — карточку с описанием что именно будет сделано. Пользователь нажимает "Подтвердить", только тогда действие выполняется.

### Шаг 6: SSE завершение
Backend шлёт событие `done` с полным ответом. Фронт показывает сообщение агента.

---

## Что происходит при следующем сообщении

LangGraph восстанавливает состояние из PostgreSQL по `thread_id`:
- История сообщений загружается автоматически
- Агент помнит контекст предыдущего разговора
- Не нужно повторять "я говорил о лососе..."

## Инструменты (Tools)

Инструменты — это функции, которые агент может вызывать для чтения или изменения данных в приложении:

| Инструмент | Что делает |
|-----------|-----------|
| `mealPlan.dayGet` | Получить план питания за день |
| `mealPlan.addEntry` | Добавить блюдо в план |
| `mealPlan.removeEntry` | Удалить блюдо |
| `recipe.create` | Создать рецепт |
| `recipe.search` | Найти рецепты |
| `shoppingList.addItem` | Добавить в список покупок |
| `selfCare.slotCreate` | Создать рутину |
| `product.search` | Найти продукт в базе |

**Политика инструментов** (AgentToolPolicyService):
Каждый режим (mode) видит только свои инструменты. Агент на странице рецептов не может случайно удалить что-то из плана питания.

## Почему LangGraph лучше чем было раньше

| Было (монолит) | Стало (LangGraph) |
|----------------|-------------------|
| Один `AgentService` с огромным switch/case | 6 специализированных узлов |
| История хранилась в памяти (терялась при рестарте) | PostgreSQL — история постоянная |
| Пользователь ждал 10-20 сек без обратной связи | SSE — видно каждый шаг в реальном времени |
| JSON из ответа парсился регулярками (хрупко) | Structured Outputs — гарантированный формат |
| Один агент отвечал за всё | Каждый агент — эксперт в своей области |

# LangSmith — мониторинг и трассировка

## Что такое LangSmith?

LangSmith — платформа от LangChain для наблюдения за работой LLM-приложений.

Представь: ты видишь каждый вызов агента как дерево:
```
Запрос пользователя "добавь яйца"
  └── RouterNode (12ms)
  └── MealPlanAgentNode (3.2s)
        └── OpenAI GPT call #1 (1.8s) — решает вызвать mealPlan.addEntry
        └── mealPlan.addEntry (0.3s) — выполняется
        └── OpenAI GPT call #2 (1.1s) — формирует ответ
```

Видно: сколько времени, сколько токенов, что пошло не так, какой был промпт.

## Текущий статус

**Не подключён.** В `.env.example` нет переменных LangSmith.

## Как подключить

### 1. Зарегистрироваться
Зайти на [smith.langchain.com](https://smith.langchain.com), создать аккаунт, создать проект "wellin".

### 2. Получить API ключ
В настройках LangSmith → API Keys → создать ключ.

### 3. Добавить переменные в `.env`
```env
LANGCHAIN_TRACING_V2=true
LANGCHAIN_ENDPOINT=https://api.smith.langchain.com
LANGCHAIN_API_KEY=ls__ваш_ключ_здесь
LANGCHAIN_PROJECT=wellin
```

### 4. Добавить в `foodieai/src/agent/langgraph/langgraph-agent.service.ts`

В начало файла (перед созданием графа):
```typescript
import { Client } from "langsmith";

// Достаточно просто поставить переменные окружения — LangGraph подхватывает автоматически
// Но можно явно: new Client({ apiKey: process.env.LANGCHAIN_API_KEY })
```

LangGraph.js автоматически отправляет трассировку если переменные окружения заданы. Дополнительного кода не нужно — просто переменные.

### 5. Добавить в `.env.example`
```env
# LangSmith (optional, for tracing)
LANGCHAIN_TRACING_V2=false
LANGCHAIN_API_KEY=
LANGCHAIN_PROJECT=wellin
```

## Что будет видно в LangSmith
- Каждый разговор пользователя с агентом
- Какие узлы графа были вызваны
- Сколько токенов потрачено
- Время ответа каждого узла
- Полные промпты и ответы OpenAI
- Ошибки с полным стектрейсом

## Когда подключать?
Рекомендую подключить до запуска — это даёт visibility в продакшене бесплатно (до определённого лимита).
