# LangGraph Migration Roadmap

## Принципы миграции

1. **Без разрыва сервиса**: каждый этап должен быть деплоируемым без downtime
2. **Обратная совместимость**: `/v1/agent/chat` API сохраняется на каждом этапе
3. **Feature flag**: переключение между старым и новым агентом через env-переменную
4. **McpService не трогаем**: инструменты переиспользуются без изменений
5. **Маленькие шаги**: каждый этап = отдельный PR с тестами

---

## Этап 0: Подготовка инфраструктуры

**Что меняется**:
- Установка LangGraph.js (`@langchain/langgraph`, `@langchain/openai`)
- Настройка LangSmith (tracing)
- Миграция БД: добавить таблицы `AgentThread`, `AgentCheckpoint`, `UserMemory`
- Заменить in-memory `AgentConversationService` на PostgreSQL-backed persistence
- Настройка pgvector extension для Semantic Memory

**Риски**:
- Миграция схемы БД в production (низкий риск — additive changes)
- Совместимость `@langchain/langgraph` с Node.js 20 + NestJS 10

**Сложность**: Средняя

**Ожидаемый результат**:
- Диалоги сохраняются в PostgreSQL (переживают рестарты)
- Трассировка всех запросов в LangSmith
- Задел для персистентности на следующих этапах

---

## Этап 1: Extraction — вынести AgentService в LangGraph без изменений поведения

**Что меняется**:
- Создать `LangGraphAgentService` — обёртка над текущей логикой в виде LangGraph graph
- Граф с одним node: `agent_node` (делает то же что `runOpenAiLoop`)
- Persistence через PostgreSQL checkpointer (LangGraph built-in)
- Feature flag `USE_LANGGRAPH_AGENT=true/false`
- Параллельно работают оба сервиса, трафик управляется флагом

**Изменения**:
```
AgentController → if USE_LANGGRAPH → LangGraphAgentService
                  else             → AgentService (старый)
```

**Риски**:
- Поведение должно быть идентичным — нужны интеграционные тесты
- LangGraph overhead (~20ms на персистентность)

**Сложность**: Средняя

**Ожидаемый результат**:
- Один кодовый путь (новый), старый — в режиме fallback
- История диалогов теперь персистентна
- Возможность трассировки в LangSmith

---

## Этап 2: Streaming — добавить SSE endpoint

**Что меняется**:
- Новый endpoint `GET /v1/agent/chat/stream` (SSE)
- Frontend использует `EventSource` вместо `fetch` для новых диалогов
- Backend стримит события: `agent_start`, `tool_call`, `tool_result`, `text_delta`, `checkpoint`, `done`
- Старый `POST /v1/agent/chat` остаётся (non-streaming fallback)

**Frontend изменения**:
- `backendAgentApi.ts` — добавить `streamBackendAgentTurn()`
- `AgentWorkspace` — показывать прогресс tool calls в реальном времени
- `AiAssistantPanel` — streaming text rendering

**Риски**:
- SSE через Cloud Run (нужна проверка timeout настроек)
- Обработка ошибок в streaming mode

**Сложность**: Высокая (много изменений на frontend)

**Ожидаемый результат**:
- Пользователь видит "Загружаю план питания..." → "Нашёл 3 продукта..." в реальном времени
- Latency воспринимается значительно меньше (первый delta через 300ms)

---

## Этап 3: Router Agent — добавить маршрутизацию

**Что меняется**:
- Новый `RouterNode` в графе: LLM принимает решение, какой агент активировать
- Граф становится multi-node: `router_node` → `{agent}_node` → `response_node`
- `AgentPromptService.buildPrompt()` реорганизован по агентам
- Удалить `switch/case` в `buildModePrompt()` — каждый агент строит свой промпт

**Примечание**: На этом этапе агентов ещё нет, Router просто выбирает между существующими mode-based промптами.

**Риски**:
- Router может маршрутизировать неправильно — нужно тщательное тестирование примерами
- Latency: лишний LLM вызов для routing (~200-400ms)

**Сложность**: Средняя

**Ожидаемый результат**:
- Архитектура готова к замене mode-based логики на независимых агентов
- Router работает, поведение системы сохранено

## Этап 4: Specialization — выделить Recipe Agent и Nutrition Agent

**Что меняется**:
- `RecipeAgent` — отдельный node со своим промптом, инструментами, structured output
- `NutritionAgent` — read-only агент для анализа рациона
- Router обучен направлять recipe/nutrition запросы к соответствующим агентам
- Переход с regex JSON parsing на OpenAI Structured Outputs для recipe draft

**Новые возможности**:
- RecipeAgent: создание черновика с гарантированной схемой (без regex)
- NutritionAgent: анализ баланса рациона за период
- RecipeAgent + NutritionAgent могут работать параллельно (LangGraph parallel nodes)

**Риски**:
- Structured Outputs может требовать изменения промптов
- Качество routing для сложных запросов (рецепт И нутриция)

**Сложность**: Средняя

**Ожидаемый результат**:
- Устранён хрупкий JSON parsing для рецептов
- Recipe creation надёжнее, быстрее, с корректными КБЖУ

---

## Этап 5: Meal Plan Agent + Shopping Agent

**Что меняется**:
- `MealPlanAgent` — отдельный node, полный контроль над meal plan operations
- `ShoppingAgent` — отдельный node для shopping list management
- Multi-agent fan-out: "составь план и сразу добавь в покупки" → оба агента параллельно
- Расширенные Human checkpoints: несколько pending actions в одном ответе

**Новые возможности**:
- MealPlanAgent: планирование на несколько дней сразу
- ShoppingAgent: генерация списка из плана питания
- Комплексный сценарий: "спланируй неделю и подготовь список покупок"

**Риски**:
- Multi-agent merging: конфликты при параллельном обновлении state
- Frontend должен обрабатывать несколько confirmations

**Сложность**: Высокая

**Ожидаемый результат**:
- Пользователь может одной командой сгенерировать недельный план
- Shopping list автоматически синхронизируется с планом

---

## Этап 6: Long-term Memory

**Что меняется**:
- `UserProfile Agent` с UserMemoryStore
- При каждом диалоге Router загружает релевантные факты о пользователе из памяти
- UserProfile Agent записывает новые предпочтения в память
- Таблица `UserMemory` (key-value + confidence + source)

**Новые возможности**:
- "Не ешь молочное" → запоминается, всегда учитывается
- "Я обычно ем омлет на завтрак" → агент не спрашивает каждый раз
- Персонализированные предложения без повторного ввода данных

**Риски**:
- Стале данные памяти (например, пользователь изменил диету — нужен механизм обновления)
- Privacy: что именно хранить, как удалять

**Сложность**: Средняя

**Ожидаемый результат**:
- Агент "знает" пользователя с первого сообщения
- Значительно меньше уточняющих вопросов

---

## Этап 7: Critic Agent + Semantic Memory

**Что меняется**:
- `CriticAgent` — финальный reviewer перед ответом пользователю
- Критик проверяет соответствие предложений ограничениям и целям
- `SemanticMemory` на pgvector: поиск похожих рецептов, исторических паттернов питания
- Router решает, когда активировать Critic (не для каждого запроса)

**Новые возможности**:
- "Этот рецепт не подходит — в нём молочное" → автоматически, не нужно объяснять
- "Что я обычно ем по понедельникам?" → semantic search по истории
- Рекомендации рецептов похожих на любимые

**Риски**:
- Critic добавляет latency (~400ms)
- Ложные срабатывания Critic могут раздражать пользователей

**Сложность**: Высокая

**Ожидаемый результат**:
- Агент персонализирован на уровне поведенческих паттернов
- Semantic search делает предложения значительно релевантнее

---

## Этап 8: Унификация frontend — удалить прямой OpenAI path

**Что меняется**:
- Удалить `openaiAgentApi.ts` и все компоненты, которые его используют
- Перевести все AI features на backend agent
- Удалить фронтенд промпты (они дублируют backend промпты)
- Единый источник истины для всей AI логики — backend LangGraph

**Риски**:
- Возможна регрессия в компонентах, которые используют direct OpenAI path
- Нужно детально проверить каждый компонент

**Сложность**: Средняя

**Ожидаемый результат**:
- Устранён технический долг
- Quota enforcement работает для всех AI вызовов
- Единое место для изменения логики агентов

## Критический путь (MVP LangGraph)

Для получения ключевых преимуществ — достаточно этапов **0 → 1 → 2 → 3**.

Это даёт:
- Персистентные диалоги (не теряются при рестарте)
- Streaming (значительно лучший UX)
- Основу для специализации

Этапы 4-7 — итеративные улучшения поверх работающей LangGraph инфраструктуры.

---

## Что НЕ меняется при миграции

- Prisma schema и все domain models
- McpService и все 31 инструмент
- REST API контракт (`POST /v1/agent/chat`)
- Auth и JWT механизм
- AI Usage / Quota система
- Все domain services (MealPlansService, RecipesService, etc.)
- Frontend UI компоненты (кроме добавления streaming)

Миграция на LangGraph = замена только **оркестрации** (AgentService + смежные сервисы). Домен остаётся нетронутым.
